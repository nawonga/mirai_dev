# Aqua-DCS Operating Rules

이 문서는 **45큐브 해수어항 DCS 운영 규칙**의 최상위 기준 문서다. 앞으로 모든 코드 제안/파일 생성/구조 변경은 이 문서를 최우선으로 따른다.

## 1. 설계 우선 원칙
- 구현보다 **구조 정의 → 역할 분리 → 인터페이스 정의**를 우선한다.
- “일단 돌아가게” 방식은 금지한다.
- 기능 추가는 **아키텍처 합의** 후 진행한다.

## 2. DCS / ESD 개념 분리
- **DCS**: 모니터링 + 제어 편의성
- **ESD**: Fail-safe, 독립적 판단
- 단일 프로세스에 모든 책임을 몰아넣지 않는다.
- ESD는 DCS와 **분리된 판단/실행 경로**를 유지한다.

## 3. 하드웨어 제약 인식
- GPIO 직접 구동 금지 (팬, 펌프, 히터 등)
- MOSFET/릴레이/드라이버 보드 등 **중간 구동 계층 필수**
- 센서 데이터는 **raw + calibrated 분리 저장**
- UPS, 전원/팬/배선 열화까지 고려한 설계를 우선한다.

## 4. 프로세스 실행 규칙
- SSH 종료 ≠ 프로세스 종료
- 모든 상시 프로세스는 아래 중 하나로 실행한다:
  - systemd
  - tmux / supervisor
- 포그라운드 상시 실행 금지

## 5. 파일 구조 변경 규칙
- 신규 폴더/파일 생성 시 반드시 문서에:
  - 목적
  - 책임 범위
  - 의존성
  을 명시한다.
- 기존 구조를 깨는 변경은 금지한다.
- 즉흥적인 파일 생성/구조 변경을 금지한다.

## 6. 데이터 관리 규칙
- DB는 로컬 기준으로 운영한다.
- Mac mini로 **주기적 백업**을 수행한다.
- 분석 결과는 원본 데이터와 **분리 저장**한다.

## 7. HMI 규칙
- 일반 모드 / 전문가 모드 분리
- 제어 UI ≠ 분석 UI
- Excel export, 그래프는 전문가 모드 전용

## 8. AI 활용 규칙
- AI는 판단 보조/요약/분석 역할에 한정한다.
- 제어 판단의 단독 주체가 되지 않는다.

# 추가 운영규칙 제안 (확장 대비)
아래는 위 규칙에 더해 장기 운영과 안정성을 위해 권장되는 추가 규칙이다.

## A. 버전/변경 이력 관리
- 아키텍처 및 데이터 스키마 변경 시 **변경 이력(CHANGELOG)** 기록
- 주요 변경 시 태그/버전 부여

## B. 센서 검증/품질 규칙
- 센서 교체/보정 시 **calibration_log 기록** 의무화
- 이상치(Outlier) 기준과 필터링 방식 명시

## C. 장애 대응 규칙
- 센서 오류, 통신 실패 발생 시 **system_events 기록**
- Fail-safe 시나리오(전원/온도/염도) 문서화

## D. 배포/운영 규칙
- 운영용과 개발용 설정 분리 (`settings.yaml` vs `settings.dev.yaml`)
- 운영 환경 변경 시 사전 점검 체크리스트 적용
